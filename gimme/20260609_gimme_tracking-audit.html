Executive read

84% Advanced means the core conversion signal is real and Smart Bidding can operate on it. The setup is more sophisticated than average — a server-side GTM container on abc.gimme.care handles the primary Google Ads firing, and two separate conversion actions cleanly separate new customers (primary, bidding signal) from all purchases (secondary, revenue reporting). That architecture is correct.

Three things need attention. The most important: both browser and server containers are actively firing Google Ads conversion tags with identical conversion labels — gads__purchase in the browser fires the same label (Z80BCMPQo6wbEJGPq9BB) as gads - purchase on the server, and same pattern for the first-time purchase tag. Google Ads deduplicates using the order_id / oid= parameter. If transaction IDs are being passed in both layers, deduplication is handling it and you are fine. If not, every purchase is counting twice — once from the browser hit, once from the server hit. This is the single highest-priority item to verify. Check via GTM Preview mode or the Google Ads Tag Diagnostics page for duplicate conversion requests on the same order ID. Second: cart data is disabled (enableProductReporting: false) on both server-side purchase tags. Gimme runs Shopping and has a Merchant Center account — turning on cart data enables product-level ROAS reporting and is a straightforward GTM change. Third: the cookie consent - gimme care tag is paused in the browser container. If Consent Mode is not active, Google cannot model conversions from users who deny cookies — this is recoverable lost signal. Unpause and verify via Chrome DevTools.

What is NOT a problem: "Purchase (GTM)" is not a legacy or redundant pixel — it intentionally tracks all purchases (new + existing customers) as a secondary reporting action, while the primary "First Time Purchase (GTM) - SST" tracks new customers only. That is a clean and deliberate setup. The url_parameters__capture_and_store and url_parameters__decorate_links tags being paused is expected — they were replaced by utm_carry. OCT diagnostics (D36-D41) are skipped by design — ecommerce tracks onsite, not CRM imports.

A fresh LP audit (2026-05-27, 13 days, 85% Good) at context/analysis/lp-audit.md flagged the intake consent banner on step 1 as inconsistent across entry paths (INT-01). This is directly connected to the paused consent tag finding above — if Consent Mode is not firing correctly, the LP audit's CVR data for the intake funnel is partly a tracking artefact, not purely page quality. Fix Consent Mode before drawing conclusions from intake CVR numbers. Re-run only if you want fresher data.

Recommendations

  1. Priority 1 — Verify browser + server deduplication (highest risk). Both gads__purchase (browser) and gads - purchase (server) fire the same conversion label Z80BCMPQo6wbEJGPq9BB. Open GTM Preview mode, complete a test purchase, and check whether both a browser-side and server-side conversion hit fire. Verify whether both include the same order_id. If no order_id is present in one or both, you are double-counting. Fix: either disable the browser-side Google Ads tags and let SST handle exclusively, or ensure order_id is passed in the browser tags for deduplication.
  2. Priority 2 — Unpause the consent tag and verify Consent Mode v2. The cookie consent - gimme care tag is PAUSED in the browser container. Unpause it, publish, then run Chrome DevTools on the landing page to verify gcs= and gcd= parameters appear in Google Ads network requests. Also directly connected to the LP audit's intake consent banner finding (INT-01) — fix this before drawing conclusions from intake CVR numbers.
  3. Priority 3 — Enable cart data on purchase tags. In the server container (abc.gimme.care), open both gads - purchase and gads - first time purchase tags and set enableProductReporting: true. Also pass an items array from the dataLayer. This enables product-level ROAS reporting in Shopping.
  4. Priority 4 — Fix counting method on secondary purchase actions (D06). "Purchase (GTM)" and "Purchase (Posthog)" both use ONE_PER_CLICK. For ecommerce purchase events these should be MANY_PER_CLICK so repeat buyers are counted correctly in those reporting columns. Change in Google Ads › Goals › Conversions › [action] › Settings › Count.
  5. Priority 5 — Fix click-through window documentation (D31). business.md says 90 days; API shows 30 days. Check Google Ads › Goals › Conversions › First Time Purchase (GTM) - SST › Settings to confirm the actual setting, then update business.md to match.
  6. Priority 6 — Rename "checkout - first time" (D04). Rename to "Checkout — First Time (GTM)" to match the naming convention used by all other actions. Minor but keeps reporting clean.
